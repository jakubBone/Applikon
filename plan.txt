  ## Zasady (Tryb Mentor)                                                   
  1. **Implementacja:** Pisz profesjonalnie (Clean Code, SOLID, Używaj      
  nowoczesnej Javy, itd.)                                                   
  2. **Krok po kroku:** Po każdym etapie zrób podsumowanie tego, co zostało 
   zrobione i dlaczego.                                                     
  3. **Edukacja:** Jeśli będzie to nowy ficzer, lub duża zmiana - tłumacz   
  mi mechanizmy działania (Java/Spring) oraz architekturę. Frontend         
  tłumaczy łopatologicznie (nie znam go), Javę tłumacz na poziomie          
  ekosystemu.                                                               
  4. **Interakcja:** Po podsumowaniu każdego kroku obowiązkowo zapytaj, czy 
   coś jeszcze wyjaśnić. Nie przechodź do kolejnego zadania bez mojego      
  wyraźnego potwierdzenia.                                                  
  5. **Zatwierdzenie:** Jeśli wprawodzamy jakąś nową funkcjonaność/refaktor 
   itp., to  implementacji i pytaniach o kod, proponujesz mi commit (ale    
  robisz go dopiero, gdy potwierdzę, ze jest ok)                            
                                                                            
  ## Konwencja commitów (muszą być po angielsku)

  └────────────────────────┴───────────────────────────────────────┘

  Aplikacja używa nowych plików, ale stare nadal siedzą w repozytorium i
  wprowadzają zamieszanie.

  Problem 2 — Jeden gigantyczny komponent (AppContent.jsx, ~800 linii)

  AppContent.jsx robi wszystko naraz:
  - zarządza stanem aplikacji (lista, formularz dodawania, formularz edycji,
  widok szczegółów)
  - obsługuje logikę API (fetch, create, update, delete, duplicate check)
  - renderuje formularze, modal edycji, widok szczegółów aplikacji
  - zawiera helper funkcje (formatDate, formatSalary, getStatusColor,
  getStatusLabel)
  - synchronizuje szerokość logo z JS (zamiast CSS!)

  To jest anty-wzorzec. Junior powinien znać zasadę Single Responsibility.

  Problem 3 — Mieszanie .jsx i .tsx

  Projekt ma TypeScript (tsconfig.json, main.tsx, hooks w .ts) — ale połowa
  komponentów to .jsx bez typów. To wygląda niechlujnie.

  Problem 4 — dist/ w repozytorium

  Folder dist/ (skompilowany build) jest na dysku, ale nie jest śledzony przez 
  gita (jest w .gitignore). To dobrze — nie ma problemu z repo. Ale warto to
  potwierdzić.

  Problem 5 — Brak "checkpoint" (snapshot funkcjonalności)

  Nie masz stanu odniesienia — żadnych testów E2E, które opisują co aplikacja 
  powinna robić, a nie jak to robi.

  Problem 6 — .vercel/project.json w repozytorium

  Plik .vercel/project.json zawiera ID projektu Vercel — to nie jest wrażliwe,
  ale nie powinno być commitowane do publicznego repo.
